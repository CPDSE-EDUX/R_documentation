R Coding Conventions

A shared style guide for pharmaceutical sciences practicals. Following these conventions makes your code readable, reproducible, and consistent across exercises.

1

Variable & Object Naming

Use snake_case for all variable names Required
Words separated by underscores — no dots, no camelCase
▶
Consistent naming makes it easy to scan code at a glance. Snake_case is the tidyverse standard and is used throughout all course materials.
✓ Do this
fill_mass_mg <- 174.2
ibu_per_mg_powder <- 0.43
label_claim_mg <- 75.0
X_bar_s1 <- mean(stage1$pct_label)
✗ Avoid this
fillMassMg <- 174.2     # camelCase
ibu.per.mg <- 0.43      # dots
lc <- 75.0              # unclear
x1 <- mean(stage1$pct_label)
Include units in variable names where relevant Recommended
Append _mg, _pct, _um, _min etc. to avoid confusion
▶
Pharmaceutical calculations often mix units (mg, µg, %, mL). Making the unit part of the name prevents subtle errors when combining variables.
✓ Do this
powder_mass_mg  <- 174.2
concentration_ug_ml <- 7.56
pct_label       <- 98.4
time_min        <- 45
✗ Avoid this
powder  <- 174.2   # mg or g?
conc    <- 7.56    # ug/mL or mg/mL?
label   <- 98.4    # % or mg?
t       <- 45      # minutes or hours?

Assignment Operator

Always use <- for assignment, never = Required
The arrow operator is the R standard and avoids confusion with function arguments
▶
Inside function calls, = sets arguments (e.g. signif(x, digits = 3)). Using <- for all assignments keeps this distinction clear and avoids hard-to-spot bugs.
✓ Do this
label_claim_mg <- 75.0
std_slope      <- 0.04478
data           <- read_csv("file.csv")
✗ Avoid this
label_claim_mg = 75.0
std_slope      = 0.04478
data           = read_csv("file.csv")
Put spaces around <- and operators Required
Space on both sides of assignment and arithmetic operators
▶
Spacing makes arithmetic readable, especially in multi-step pharmaceutical calculations where it's easy to misread a minus sign.
✓ Do this
C2 <- (absorbance - std_intercept) / std_slope
AV <- k * S
fill_mass_mg <- mass_filled_mg - mass_empty_mg
✗ Avoid this
C2<-(absorbance-std_intercept)/std_slope
AV<-k*S
fill_mass_mg<-mass_filled_mg-mass_empty_mg

Printing Results

Use cat() to print labelled results Required
Always include a descriptive label and unit — never print a bare number
▶
A bare number printed to the console has no context. In pharmaceutical reporting, always state what the number means and what unit it is in. cat() lets you compose a readable sentence from text and values.
✓ Do this
cat("Ibuprofen content:", 
    signif(ibu_per_mg_powder, 3), 
    "mg Ibuprofen per mg powder\n")

cat("Mean content (X̄):", signif(X_bar, 4), 
    "% of label claim\n")
✗ Avoid this
ibu_per_mg_powder   # just the variable
print(ibu_per_mg_powder)  # no label
X_bar               # no context
Use view() to inspect data frames Required
Opens the table viewer — do not print large data frames to the console
▶
Printing a 30-row data frame floods the console output and makes the document hard to read when rendered. Use view(data) to inspect it interactively, and head(data) if you need a preview in the document output.
✓ Do this
view(data)        # interactive inspection
head(data, 5)     # first 5 rows in output
✗ Avoid this
print(data)       # floods the output
data              # dumps entire table

Rounding & Significant Figures

Use signif() for final results, round() for fixed decimals Required
Only round at the point of printing — keep full precision in intermediate calculations
▶
Rounding intermediate values accumulates error. Store full-precision values in variables and only apply rounding when displaying the result. Use signif(x, n) when the number of significant figures matters (most analytical results), and round(x, n) when a fixed number of decimal places is needed (e.g. always show 2 decimal places for %).
Example
# Store full precision — do NOT round here
C2 <- (absorbance - std_intercept) / std_slope
ibu_per_mg <- C2 * 20 / 0.1 * 50 / 1000 / powder_mass_mg

# Only round when printing
cat("Ibuprofen per mg powder:", signif(ibu_per_mg, 3), "mg/mg\n")
cat("Label claim %:", round(pct_label, 2), "%\n")
Never round inside a mutate() column Recommended
Rounding a column used in later calculations propagates the error
▶
If you round fill_mass_mg inside mutate and then use it to compute ibu_mg, you introduce a rounding error that compounds. Keep data columns at full precision; round only in cat() when reporting.
✓ Do this
data %>%
  mutate(
    fill_mass_mg = mass_filled_mg - mass_empty_mg,
    ibu_mg       = fill_mass_mg * ibu_per_mg_powder
  )
✗ Avoid this
data %>%
  mutate(
    fill_mass_mg = round(mass_filled_mg - mass_empty_mg, 1),
    ibu_mg       = fill_mass_mg * ibu_per_mg_powder
  )

Pipes & Data Manipulation

Use %>% with each step on its own line Required
One function per line, indented 2 spaces, pipe at end of line
▶
Multi-step data manipulation written as a single long line is very hard to read and debug. Putting each step on its own line makes it easy to comment out a single step or add a new one.
✓ Do this
data <- data %>%
  mutate(
    fill_mass_mg = mass_filled_mg - mass_empty_mg,
    ibu_mg       = fill_mass_mg * ibu_per_mg_powder,
    pct_label    = ibu_mg / label_claim_mg * 100
  )
✗ Avoid this
data <- mutate(data, fill_mass_mg=mass_filled_mg-mass_empty_mg, ibu_mg=fill_mass_mg*ibu_per_mg_powder, pct_label=ibu_mg/label_claim_mg*100)
Group constants at the top of a code chunk Recommended
Define all "magic numbers" as named constants before using them
▶
Grouping constants at the top means students only need to change values in one place. It also makes the intent of the code clearer — it's obvious that vol_flask1_mL is a setup parameter, not a computed result.
✓ Do this
# Constants — update these with your values
vol_flask1_mL <- 50.0
vol_aliquot_mL <- 0.1
vol_flask2_mL <- 20.0

C1 <- C2 * vol_flask2_mL / vol_aliquot_mL
✗ Avoid this
C1 <- C2 * 20 / 0.1  # where do 20 and 0.1 come from?

Plotting with ggplot2

Always use theme_classic() as the base theme Required
Clean white background, no gridlines — appropriate for scientific figures
▶
theme_classic() produces figures that closely match the style expected in scientific reports and publications — white background, visible axis lines, no distracting grid. It is used consistently across all course exercises.
Standard template
ggplot(data, aes(x = time_min, y = mean_pct)) +
  geom_line() +
  geom_point() +
  labs(
    x     = "Time (min)",
    y     = "Dissolved (%)",
    title = "Dissolution profile — Ibuprofen 75 mg capsules"
  ) +
  theme_classic()
Always label axes with units in parentheses Required
Use labs(x = "...", y = "...") — never leave default variable names as axis labels
▶
Default ggplot axis labels show the raw column name (e.g. mean_pct), which is unacceptable in a scientific figure. Always write a full, human-readable label with units.
✓ Do this
labs(
  x = "Time (min)",
  y = "Ibuprofen released (mg)"
)
✗ Avoid this
# No labs() call — axis shows "time_min"
# and "mean_mg" — not acceptable
Each + layer goes on its own line Required
One geom or modifier per line, + at end of the preceding line
▶
This mirrors the pipe convention for data manipulation. It makes it easy to comment out a layer (e.g. temporarily remove error bars) without breaking the rest of the plot code.
✓ Do this
ggplot(data, aes(x = time_min, y = mean_mg)) +
  geom_line() +
  geom_point() +
  geom_errorbar(aes(ymin = mean_mg - sd_mg,
                     ymax = mean_mg + sd_mg)) +
  theme_classic()
✗ Avoid this
ggplot(data,aes(x=time_min,y=mean_mg))+geom_line()+geom_point()+theme_classic()
Use "steelblue" as the default single-series colour Recommended
For multi-group plots, let ggplot assign colours automatically via aes(color = group)
▶
A consistent default colour keeps all course figures looking coherent. For single-series plots, "steelblue" is readable on screen and prints well in greyscale. For multi-series plots, map the grouping variable to color inside aes() and let ggplot handle the palette.
✓ Single series
geom_line(color = "steelblue")
geom_point(color = "steelblue")
✓ Multiple groups
ggplot(data, aes(x = time_min, y = mean_pct,
              color = formulation)) +
  geom_line() +
  geom_point()

Code Comments

Comment why, not what — and always comment units Required
A comment should add information, not just repeat what the code already says
▶
Restating the code in English adds no value. Useful comments explain the reasoning, the source (e.g. which EP table), or the unit of a variable.
✓ Useful comments
label_claim_mg <- 75.0  # mg — labelled dose per capsule
k_s1 <- 2.4            # EP 2.9.40, Table 2.9.40-1, n=10
L1   <- 15.0           # acceptance limit, Stage 1 (%)
✗ Unhelpful comments
label_claim_mg <- 75.0  # set label claim to 75
k_s1 <- 2.4            # k value
L1   <- 15.0           # L1
Use section separator comments in long scripts Recommended
Group related lines with a # --- header comment
▶
Example
# --- Assay: Ibuprofen concentration from UV absorbance ---
C2 <- (absorbance - std_intercept) / std_slope
C1 <- C2 * vol_flask2_mL / vol_aliquot_mL

# --- EP 2.9.40 Stage 1 acceptance value ---
AV_s1 <- k_s1 * S_s1
